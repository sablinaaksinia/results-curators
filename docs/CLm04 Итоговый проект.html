1. Зачем нужен финальный проект

Цель финального проекта ↘️

Финальный урок курса по Claude Code посвящён созданию проекта для портфолио.

Финальный проект — это демонстрация знаний и навыков через решение реальной задачи. Он показывает, что вы не просто смотрели уроки, а научились создавать полноценные рабочие инструменты.

Финальный проект — это не отписка ради сертификата, а демонстрация ваших навыков через реальную задачу.

Проект должен быть полезен лично вам: автоматизировать рутину, ускорить рабочие процессы, освободить время для более важных задач. Лучший ориентир — рабочие задачи, потому что именно в них скрыт наибольший потенциал для автоматизации с помощью вайб-кодинга.

Что даёт финальный проект ↘️

Завершённый проект решает сразу несколько задач:

  • становится первым кейсом в портфолио (или очередным — если вы уже создавали проекты в ходе курса)
  • может служить основой для фриланс-услуг или внутренней автоматизации
  • демонстрирует заказчикам и клиентам ваш реальный уровень
  • помогает ускорить и улучшить рабочие процессы, возможно — сократить рабочий день
Формат проекта — MVP ↘️

Мы создаём не масштабный продукт, а MVP — минимально жизнеспособный продукт. Это версия проекта, которая уже функционирует и решает вашу задачу, но может иметь нюансы для доработки в будущем.

На создание финального проекта отводится две недели: этого достаточно, чтобы реализовать, продемонстрировать проект и получить сертификат завершения обучения.

2. Как выбрать идею для проекта

Шаг 1 — Найти повторяющуюся задачу ↘️

Если идея пока не появилась — это нормально. Воспользуемся простым подходом: выпишем все свои рабочие задачи и разберём их по категориям:

  • задачи, которые выполняются легко и без напряжения
  • задачи, которые повторяются — ежемесячно, еженедельно, ежеквартально
  • задачи, которые отнимают много времени, являются рутинными и вызывают сложности

Именно задачи из третьей категории — лучшие кандидаты для автоматизации с помощью вайб-кодинга.

Пример из практики: сбор и анализ статистики по курсам — анкеты из разных источников, отзывы, данные из чатов — раньше занимал целый рабочий день и повторялся каждый квартал. После автоматизации с помощью вайб-кодинга та же задача занимает 1–2 часа, а отчёт готов сразу — можно немедленно приступать к улучшению курса.

Проект рождается не из «что бы мне такое придумать», а из «какую реальную задачу я могу упростить с помощью Claude Code?»
Шаг 2 — Сформулировать идею и ограничить её до MVP ↘️

После того как задача определена, формулируем идею сервиса: что делалось вручную — теперь будет автоматически.

Важно ограничить проект до MVP: не «платформа для маркетологов», а «сервис для автогенерации постов в одной конкретной социальной сети». Сужение идеи помогает точно уложиться в две недели и получить завершённый, стабильно работающий результат.

Шаг 3 — Проверить идею по критериям ↘️

Хороший финальный проект соответствует всем этим критериям:

  • Понятный — задачи, итоги и польза проекта очевидны с первого взгляда
  • Полезный — решает реальную задачу для вас или вашей команды
  • Завершённый — ограниченный набор функций, которые работают стабильно
  • Демонстрируемый — можно показать скриншоты, результаты, провести демо без багов
  • Реальный — взят из личного опыта или рабочей практики, а не выдуман
  • Выполнимый за две недели — с учётом занятости и других задач
📜 Идеи выпускного проекта для курса по Claude Code ↘️

Выбери одну идею из списка или придумай свою. Каждый проект должен быть небольшим MVP: понятная задача, простой интерфейс, рабочий результат и возможность показать проект через скриншоты или GitHub.

3. Как оформить описание проекта

Структура описания проекта ↘️

Проект нужно не только создать, но и правильно оформить. Описание проекта строится по следующей структуре:

  • Название — короткое, ёмкое, сразу понятное. Например: «Автогенератор презентаций по шаблону»
  • Проблема — какую задачу решает проект и почему она важна
  • Пользователь — для кого создаётся проект (методисты, маркетологи, команда, конкретная аудитория)
  • Основная функция — что именно делает сервис, по пунктам: что подаётся на вход, что получается на выходе, какие инструменты задействованы
  • Формат результата — что получит пользователь: готовая презентация, отчёт, статистика
  • Минимальный функционал — что будет реализовано в первой версии (MVP); здесь же можно отметить, что планируется добавить в будущем
  • Технологии — какой язык программирования использован, какие библиотеки, форматы ответов
Описание как инструмент разработки ↘️

Подробное описание с болями пользователя, ожидаемыми результатами и ограничениями — это не просто документация. Это контекст для Claude Code, который позволяет ему с первого промпта адаптировать проект под конкретные задачи. Чем подробнее описание, тем точнее и качественнее будет результат разработки.

Структура описания:

  • Название— короткое и понятное
  • Проблема— какую задачу решает проект
  • Пользователь— для кого создаётся
  • Основная функция— что сервис должен делать
  • Формат результата— что видит пользователь на выходе
  • Минимальный функционал— что должно быть в первой версии
  • Технологии— Python, Flask, HTML/CSS и т.д.

4. Как разрабатывать проект

Этапы разработки ↘️

Процесс создания финального проекта выглядит так:

  1. Создаём рабочую папку — называем её на английском языке (русский текст в названии может вызвать ошибки в Claude Code)
  1. Просим Claude Code составить план и разработать структуру проекта
  1. Набрасываем код и создаём MVP
  1. Запускаем локально и проверяем работу
  1. Дорабатываем: улучшаем интерфейс, тексты, добавляем функционал
  1. Готовим README-инструкцию
  1. При желании — публикуем проект на виртуальном сервере
Локально vs. публикация ↘️

Проект может работать локально — этого достаточно для финального проекта. В этом случае демонстрация проходит через скриншоты каждого шага.

Публикация на виртуальном сервере не обязательна, но значительно усиливает впечатление: вместо скриншотов можно прислать ссылку, по которой проект можно открыть, покликать и взаимодействовать с ним в реальном времени.

Для портфолио важно не только то, что проект существует, но и то, что его можно показать: как он выглядит, какой у него интерфейс, какие даёт результаты.

5. Критерии хорошего финального проекта

Что проверяется в итоговом проекте ↘️

В вайб-кодинге руки развязаны — нет жёстких технических требований вроде «обязательно использовать две нейросети». Главное — чтобы проект решал реальную задачу. Тем не менее, есть ряд критериев, на которые стоит ориентироваться:

  • Понятная идея — сразу ясно, что делает проект и зачем он нужен
  • Рабочий локальный запуск — проект стабильно работает, каждый шаг можно продемонстрировать скриншотом
  • Минимальный интерфейс — есть визуальная оболочка для взаимодействия
  • README с инструкцией запуска — описание проекта и пользовательский сценарий
  • Демонстрируемость — можно показать, как проект работает, без слов «ну, это предположительно работает»
  • Чистая структура файлов — нет лишних файлов и «мусора» в коде
  • Ссылка на GitHub-репозиторий — проект опубликован и доступен
Скилл проекта ↘️

Помимо репозитория, стоит создать скилл для своего проекта — специальный файл, с помощью которого коллеги смогут развернуть точно такой же проект у себя. Это удобный способ делиться работой внутри команды.

Сильный проект — не самый сложный, а тот, который можно открыть, понять, запустить и показать.

6. GitHub — оформление профиля и публикация проекта

Зачем нужен GitHub ↘️

Мы создаём профиль на GitHub, публикуем туда финальный проект и оформляем его так, чтобы любой, кто зайдёт на страницу, сразу понял: кто вы, что умеете и что создали.

Создание и оформление профиля ↘️
  1. Переходим на github.com → нажимаем Sign up for GitHub → проходим регистрацию
  1. Нажимаем на иконку профиля → Profile
  1. Edit profile
  1. Заполняем: имя (лучше на английском), должность или роль, компанию, локацию, email, ссылки на соцсети
  1. Загружаем аватар (если фото слишком большое — сжимаем через онлайн-сервис сжатия изображений)
  1. Сохраняем изменения
README-файл профиля ↘️

Чтобы профиль стал запоминающимся, добавляем README-файл — он отображается прямо на главной странице профиля.

Как создать:

  1. Копируем своё имя пользователя (username) на GitHub
  1. Создаём новый репозиторий с тем же именем, что и username
  1. Переводим репозиторий в Public, добавляем README-файл → Create repository
  1. Переходим в Claude и просим сгенерировать README в формате Markdown с описанием профиля: чем занимаетесь, какие проекты делаете, ваш стек
  1. Копируем готовый Markdown-текст → вставляем в README-файл репозитория → Commit changes

7. Публикация проекта на GitHub через GitHub CLI

Установка и авторизация GitHub CLI ↘️

Чтобы переносить локальные файлы в GitHub прямо с компьютера, нужен GitHub CLI (gh).

  1. Переходим на страницу GitHub CLI → скачиваем версию для своей операционной системы → устанавливаем (Next → Next → Install)
  1. Открываем терминал (командную строку)
  1. Вводим команду для авторизации: gh auth login
  1. Выбираем GitHub.com → HTTPS → подтверждаем (`Y`)
  1. Следуем инструкциям: вводим код из терминала в браузере для подтверждения
  1. После успешной авторизации Claude Code получает доступ к GitHub
Публикация проекта ↘️
  1. Открываем папку с проектом в Claude Code
  1. Просим Claude Code опубликовать проект из этой папки в аккаунт GitHub как новый репозиторий и написать README.md с подробным описанием
  1. В промпт можно добавить всю структуру описания проекта (название, проблема, пользователь и т.д.)
  1. Claude Code создаёт README-файл и публикует репозиторий
  1. Переходим на GitHub → раздел Repositories → видим опубликованный проект
Добавление скриншотов в README ↘️
  1. На странице репозитория нажимаем Add file → Create new file
  1. В имени файла пишем images/ — это создаёт папку для скриншотов
  1. Также пишем название файла (любое). Commit changes
  1. В папке images нажимаем Add file → Upload files → загружаем скриншоты проекта
  1. Commit changes — сохраняем
  1. Переходим в README-файл → нажимаем карандаш (редактировать)
  1. Добавляем изображение в Markdown-синтаксисе: ![описание картинки](images/имя_файла.png)
  1. Commit changes

8. Создание и публикация скилла проекта

Что такое скилл и зачем он нужен ↘️

Скилл описывает: структуру репозитория, как запустить сервис, как вносить изменения, как проверять ошибки, как готовить проект к публикации.

Как создать и опубликовать скилл ↘️
  1. Открываем командную строку. Переходим в Claude Code. Переходим в папку с проектом в Claude Code
  1. Используем промпт: «Создай скилл для моего проекта. Он должен помогать Claude Code работать с этим репозиторием: понимать структуру, запускать сервис, вносить изменения, обновлять README, проверять ошибки и готовить проект к публикации»
  1. Claude Code создаёт папку `.claude/` со скиллом внутри
  1. Просим Claude Code опубликовать папку со скиллом в тот же репозиторий на GitHub
  1. Проверяем: в репозитории появляется раздел с готовым скиллом

После публикации финальный репозиторий содержит всё необходимое:

  • подробное README с описанием проекта
  • скриншоты работающего проекта
  • скилл для передачи коллегам
  • ссылку на опубликованный проект (если есть)

9. Онлайн-выпускной и защита проекта

Формат онлайн-выпускного ↘️

После двух недель работы над проектом проходит онлайн-выпускной, где каждый желающий может представить свою работу. Выступление занимает 3–5 минут.

Структура презентации проекта:

  • Название и короткое описание — что это за проект
  • Проблема — какую задачу вы решили
  • Живая демонстрация — показываем, как проект работает (даже если он локальный)
  • Результаты — что получается на выходе
  • GitHub — показываем репозиторий
  • Планы — как хотите развивать проект дальше или какие следующие проекты планируете реализовать с помощью вайб-кодинга

10. Итоги

Дополнительные материалы

Ссылки на сервисы из урока ↘️
  • GitHub (github.com) — платформа для хранения и публикации репозиториев с кодом, документацией и скриншотами проектов
  • GitHub CLI (cli.github.com) — утилита командной строки для работы с GitHub прямо с локального компьютера
  • Claude / Claude Code (claude.ai) — AI-ассистент, используемый для разработки проектов, создания README, скиллов и оформления профиля
  • Онлайн-сервис сжатия изображений — любой удобный сервис (например, squoosh.app или tinypng.com) для уменьшения размера фотографий перед загрузкой на GitHub
Словарь занятия ↘️
  • MVP (Minimum Viable Product) — минимально жизнеспособный продукт; версия проекта с ограниченным, но работающим функционалом, которую можно оценить и использовать
  • Вайб-кодинг — подход к разработке с помощью AI-инструментов (в первую очередь Claude Code), при котором код пишется и дорабатывается через диалог с ассистентом
  • Репозиторий — хранилище файлов проекта на GitHub (код, README, скриншоты, скилл)
  • README.md — текстовый файл в формате Markdown с описанием проекта: название, задача, инструкция по запуску, скриншоты
  • Скилл (Claude skill) — файл-инструкция для Claude Code, позволяющий воспроизвести проект в новом окружении; удобный способ делиться проектом с коллегами
  • GitHub CLI — инструмент командной строки для управления репозиториями GitHub с локального компьютера
  • Markdown — язык разметки текста, используемый для оформления README и других документов на GitHub
  • Портфолио — набор завершённых проектов, которые демонстрируют навыки и опыт разработчика потенциальным работодателям или заказчикам
  • Виртуальный сервер — удалённая среда для публикации проекта, чтобы им можно было пользоваться через браузер без локального запуска

Название
Telegram-бот для управления командой преподавателей онлайн-школы «Матриус»

---
Проблема
Два процесса отнимали время куратора каждую неделю:
1. Вручную составлять и отправлять в чат отчёт о нарушениях дисциплины — кто не поставил статус, кто не добавил ссылку, кто отменил уроки по своей инициативе
2. Отвечать преподавателям на однотипные вопросы по регламентам — в любое время дня, включая вечер и выходные

Оба процесса нужно было автоматизировать, сохранив контроль над качеством данных и ответов.

---
Пользователь
Преподаватели онлайн-школы «Матриус» (~80 человек) и куратор группы. Общение происходит в рабочем Telegram-чате.

---
Основная функция

Компонент 1 — Еженедельный отчёт о дисциплине:
- Вход: Excel-файл с данными из CRM, загруженный в папку data/
- Обработка: watcher отслеживает появление нового файла, парсит таблицу, формирует 5 сообщений — нарушения, отмены, похвала за дисциплину
- Выход: серия сообщений в Telegram-чат — кто нарушил и что именно, кто сработал без замечаний, кто идеален по всем показателям

Компонент 2 — QA-бот по базе знаний:
- Вход: сообщение в чат, начинающееся с ? — например, ? как перенести урок
- Обработка: поиск по Markdown-регламентам (стемминг + синонимы), передача контекста в языковую модель Groq LLaMA
- Выход: ответ с конкретной инструкцией, источником и ссылкой на Tilda

---
Формат результата
- Отчёт о дисциплине: 5 структурированных Telegram-сообщений с именами, цифрами и эмодзи-разметкой
- Ответ на вопрос: реплай в чат с инструкцией, названием документа (📖) и ссылкой на регламент (🔗); если ответа нет — явно направляет к куратору

---
Минимальный функционал (MVP)

Реализовано:
- Watcher следит за папкой data/ и запускает отправку автоматически при появлении нового файла
- Парсинг Excel через pandas, подсчёт нарушений по 7 показателям
- 5 типов сообщений: нарушения, отмены по инициативе тренера, похвала за отсутствие отмен, похвала за лояльных учеников, топ без нарушений
- Long polling Telegram-чата, реакция на сообщения с ?
- Keyword-поиск по MD-файлам базы знаний с фильтрацией ученических секций
- Генерация ответа через Groq API (LLaMA 3.3-70b-versatile)
- Fallback-режим без AI

Планируется:
- Семантический поиск (embeddings) вместо keyword
- Автообновление базы знаний при изменении файлов на Tilda
- Статистика: какие вопросы задают чаще всего

---
Технологии
- Язык: Python 3
- Telegram Bot API — long polling и sendMessage с HTML-разметкой
- Groq API — модель llama-3.3-70b-versatile
- pandas — парсинг Excel с данными дисциплины
- База знаний — Markdown-файлы в project-knowledge-base/matrius-docs/
- Windows Task Scheduler — ежедневный запуск watcher в 7:00
- Конфигурация — .env-файл
